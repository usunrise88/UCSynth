# Как писать железные гайды

Читать при работе с периферией. Пользователь знает электронику — **не объясняй основы**. Нужен исполнимый гайд, не теория.

---

## Структура гайда — 4 блока

### 1. Подключение — таблица пин-в-пин

Не «подключи по I2C», а конкретно:

```
VL53L0X  →  ESP32-S3
VIN      →  3.3V
GND      →  GND
SDA      →  GPIO 8    (общая I2C-шина)
SCL      →  GPIO 9    (общая I2C-шина)
XSHUT    →  GPIO 10   (только если два сенсора)
```

Проверь по `progress.md`, что пины не конфликтуют с уже занятыми.

### 2. Грабли этого конкретного модуля

Только то, что **реально ломает работу**:

> - **SCK → GND** (иначе тишина — причина №1)
> - **XSMT → 3.3V** (иначе soft-mute)

**Не писать:** «не перепутай питание и землю», «I2C — двухпроводная шина, где...».

### 3. Проверка до кода

Что померить мультиметром / посмотреть анализатором **до** запуска прошивки:

```
До запуска I2S:
- Мультиметр: XSMT ≈ 3.3V при поданном питании (не 0)
- Мультиметр: SCK звонится на GND
- I2C-сканер: устройство видно по адресу 0x29
```

### 4. Диагностика — дерево «симптом → проверить»

От самого вероятного к редкому:

```
Тишина:
1. SCK на GND?                    ← причина №1
2. XSMT не в мьюте?
3. Анализатор: идут ли BCK/LCK?
   → нет:  проблема в конфиге I2S
   → да, но нет данных на DIN: проблема в DMA/буфере
4. Питание VIN есть?
```

---

## Общие правила

- **Обновляй карту пинов** в `progress.md` при подключении новой периферии
- Следи за **strapping-пинами S3** — не занимать критичным
- Аналоговое питание (PCM5102) — **отдельно от цифрового**
- Новая периферия на I2C — **прогони сканер**, проверь адрес на конфликт

---

## Чего НЕ писать

- Теории про протоколы
- Общих правил безопасности
- Объяснений, зачем конденсатор по питанию — просто скажи какой и куда
- Того, что верно для любого модуля вообще
