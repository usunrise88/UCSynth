# План разработки: от бредборда к синту

Спецификация: `spec.md`. Железо: `hardware.md`. Риски: `risks.md`.

**Принципы:**
- На каждой вехе — **работающий результат**, не «полуфабрикат»
- Управление (энкодеры/матрица) и корпусирование — **в конце**
- GUI-пульт — **в начале**, он заменяет физические контролы на всё время разработки
- Риск и неопределённость — вперёд, рутина — назад

---

## ЭТАП 0. Каркас и протокол

### 0.1 Скелет проекта
- PlatformIO + ESP-IDF, таргет ESP32-S3-N16R8
- **Проверить**: поддержка нужной версии IDF в PlatformIO, работа PSRAM
- Структура: `audio/` (DSP), `control/` (параметры), `io/` (периферия), `comm/` (serial)
- Workflow облако→локально: сборка, скачивание бинаря, прошивка esptool

**Результат:** проект собирается, «Hello» в serial, PSRAM определяется.

### 0.2 Модель параметров (ядро архитектуры)
Центральная вещь всего проекта. Спроектировать **до** написания DSP.

- Реестр параметров: `id → {имя, тип, диапазон, значение, дефолт}`
- API: `set_param(id, value)` / `get_param(id)`
- **Все источники управления пишут только через этот API** (GUI, энкодеры, тач)
- Thread-safe: пишет Core 1, читает Core 0 (аудио)

**Результат:** параметры создаются, читаются, меняются.

### 0.3 Протокол Serial (USB CDC)
- Бинарный или текстовый (текстовый проще отлаживать, бинарный быстрее — **рекомендую текстовый на старте**)
- Команды: `GET <id>`, `SET <id> <val>`, `LIST` (дамп реестра), `NOTE_ON <n> <vel>`, `NOTE_OFF <n>`, `STAT` (CPU/RAM/underruns)
- Дамп реестра позволяет GUI **строить интерфейс динамически** — не хардкодить параметры

**Результат:** через терминал можно менять параметры и видеть их.

## ЭТАП 3. Голос целиком

### 3.1 ADSR (VCA)
Экспоненциальные сегменты. **Gate-driven** — огибающая читает *gate*, а не событие note (чтобы latch/drone был просто «держим gate»). Триггер по NOTE_ON/OFF.
**Drone (заложить сразу):** latch — note-on фиксирует gate, sustain держится бесконечно до явного стопа; опц. зацикливание огибающей (cycling) для эволюции дрона.
**Результат:** ноты с атакой/затуханием; latch держит дрон.

### 3.2 Фильтр (SVF)
Cutoff, resonance, режимы LP/HP/BP. + ADSR для фильтра.
**Результат:** звук «оформился».

### 3.3 Три осциллятора + микшер
3 wavetable-осц, детюн, уровни, шум, ring mod.
**Результат:** богатый тембр.

### 3.4 Lo-fi режим осциллятора
Bit-reduction до 8 бит, отключение антиалиасинга.
**Результат:** второй характер звука почти бесплатно.

### 3.5 Полифония
Voice allocation (round-robin / oldest-steal), 1 → 4 → 8 голосов.
**Здесь измеряем CPU.** Если не тянет 8 — понижаем или упрощаем.
**Результат:** ✅ **Играет аккорды. Известен потолок полифонии.**

### 3.6 Glide / Portamento
Плавное скольжение высоты между нотами (параметр времени). Mono-glide и poly-glide (по ближайшему голосу); legato-режим (не ретригерить огибающую).
**Результат:** плавные переходы высоты; mono-лид и дрон-глайд.

## ЭТАП 4. Модуляция

### 4.1 LFO ×2 — 🔨 собран (замер CPU на железе за пользователем)
Формы волн (sine/tri/saw/square/S&H), скорость, control-rate. Реализован вместе с **ядром мод-матрицы**
(8 гибких слотов src→dst с глубиной, `voice.cpp`) — LFO без приёмника не проверить. Источники: LFO1/2,
VCF-огиб., velocity, mod-wheel. Приёмники: pitch, cutoff, resonance, amp.

### 4.2 Wave-огибающая (8 точек, зацикливаемая)
Гуляет по позиции в wavetable. **Ключевая фича характера.**

### 4.3 Матрица модуляции
Источники → приёмники, глубина. Роутинг из GUI.

**Результат:** ✅ **Синт «ожил»** — эволюционирующие тембры.

## ЭТАП 5. Эффекты

Порядок — от дешёвого к дорогому, ревер последним (максимальный риск по CPU).

### 5.1 Overdrive
Waveshaper (tanh/полином). Без оверсэмплинга. Правильный gain staging на входе *(урок Analog Heat: аттенюация для headroom)*.

### 5.2 Delay
Кольцевой буфер **в PSRAM**. Feedback, фильтр в цепи ОС, tempo-sync позже.

### 5.3 Reverb
Freeverb/Schroeder, буферы **в PSRAM**.
**Здесь узнаем, сколько осталось CPU.** Если не хватает — упрощаем ревер или режем полифонию.

**Результат:** ✅ **Полный звуковой тракт готов.** Дальше — только управление.

## ЭТАП 7. Секвенсор и арпеджиатор

Общий темп-клок (bpm, play/stop, **swing**) — на нём работают и секвенсор, и арпеджиатор.

### 7.1 Step-секвенсор
16 шагов. Пока управляется из GUI.

### 7.2 P-locks *(ключевая фича Elektron)*
Значение параметра, привязанное к шагу. Структура данных шага.
**Заложить в модель данных с самого начала этапа.**

### 7.3 Арпеджиатор
Из удержанных нот: **up / down / random**, диапазон (октавы), скорость (деление клока), **hold/latch** (дружит с drone/glide).

### 7.4 Дополнительно
Trig probability (вероятность срабатывания шага).

**Результат:** ✅ **Инструмент, а не тон-генератор.**

## ЭТАП 8. Физическое управление

*Только сейчас — когда звук и логика готовы и отлажены через GUI.*

### 8.1 I2C-шина
- Подключить MCP23017 ×3, развести адреса (0x20/0x21/0x22)
- **Убрать pull-up на двух платах**
- I2C-сканер → все устройства видны

### 8.2 Матрица 4×4
Кнопки + диоды 1N4148, дебаунс. → NOTE_ON/OFF через **ту же модель параметров**.

### 8.3 Энкодеры
Чтение через INT (ITA/ITB, mirror mode), таблица квадратур, дебаунс.
Опционально: активные — на PCNT S3.
Маппинг: энкодер → параметр (через тот же API, что и GUI).

### 8.4 Кнопки и Shift-слой
Play/Pause, Menu, Shift. Второй слой для всех контролов.

### 8.5 Режимная модель
note-mode / seq-mode / p-lock / menu. State machine.

**Результат:** ✅ **Синтом можно играть без компьютера.**

## ЭТАП 12. Дополнительные типы осцилляторов

*Порядок условен: это DSP-расширение голоса (этап 3), делается после освоения базового звука.
Вынесено в конец, чтобы не перенумеровывать этапы 6–11. Каждый тип — отдельно, с замером CPU.*

Гибридная модель (см. `spec.md`): в Classic-движке тип на слот; FM и Karplus-Strong — отдельные движки голоса.

### 12.1 VirtualAnalog (PolyBLEP)
Пила/меандр/треугольник без алиасинга (PolyBLEP-коррекция разрывов). Тип слота в Classic.

### 12.2 Phase Distortion
Casio CZ-стиль: фаза читается через warping-функцию перед таблицей. Тип слота в Classic.

### 12.3 FM (2 оператора)
Несущая + модулятор, индекс и отношение частот. Отдельный движок голоса.

### 12.4 Karplus-Strong
Щипковая струна: короткое возбуждение (шум) → линия задержки с фильтром в ОС. Отдельный движок; буфер задержки короткий, во внутренней RAM.

**Результат:** ✅ **Палитра из пяти типов осцилляторов.**

## Почему такой порядок

1. **GUI рано** → отлаживаешь DSP без единого энкодера. Не блокируешься на железе.
2. **Модель параметров первой** → энкодеры и тач подключатся позже без переписывания движка.
3. **Звук до управления** → если DSP не взлетит, не потратишь время на панель.
4. **Reverb перед управлением** → узнаешь потолок CPU до того, как добавишь UI-нагрузку.
5. **Корпус последним** → размеры и раскладка станут ясны только когда всё работает.
