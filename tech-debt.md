## Активный долг

### [T-001] storage-раздел: файловая система не выбрана
- **Где:** `partitions.csv`
- **Что:** раздел `storage` помечен `spiffs` (8 МБ), но ФС не финализирована (spiffs / littlefs / fat)
- **Почему:** на этапе 0 неизвестен паттерн доступа (wavetable-банки — только чтение? пресеты — запись мелкими файлами, износ flash?)
- **Цена:** смена подтипа/раскладки позже = переразбивка flash, потеря содержимого раздела
- **Когда чинить:** этап 1 (wavetable-банк) и этап 6 (пресеты) — тогда выбрать ФС под реальный доступ

### [T-002] Консоль по умолчанию — USB-Serial-JTAG, не UART
- **Где:** `sdkconfig.defaults` → `CONFIG_ESP_CONSOLE_USB_SERIAL_JTAG`
- **Что:** логи идут в нативный USB S3, а не в UART-мост
- **Почему:** проект целится в USB CDC; дешёвые N16R8-платы часто выводят только нативный USB; прошивка и лог — по одному кабелю
- **Цена:** если плата подключена через UART-мост (CP2102/CH340), на первом флэше логов не будет, пока не переключить консоль на UART0
- **Когда чинить:** на первом запуске на железе — если в мониторе тишина, см. `docs/build-flash.md` → «нет логов»

### [T-003] Логи и протокол — по одному каналу USB-JTAG
- **Где:** `components/comm/`, `docs/serial-protocol.md`
- **Что:** `ESP_LOG` (ASCII) и бинарные кадры протокола идут по одному USB-Serial-JTAG
- **Почему:** дешёвые платы дают один нативный USB; отдельный канал усложнил бы «один кабель»
- **Цена:** нужен демукс — кадры несут sync `55 AA` + CRC; при смешивании байтов лога с кадром CRC не сойдётся и кадр теряется (GUI перезапросит). Редко, но возможно.
- **Когда чинить:** если поток кадров станет плотным/потери заметны — увести логи на UART0 (порт CH343), протокол оставить на нативном USB. Пока не требуется.

### [T-004] I2C-шина принадлежит компоненту `display`
- **Где:** `components/display/src/display.cpp` (`display_init`)
- **Что:** master-шину I2C (SDA=8/SCL=9) создаёт `display`, хотя шина общая (позже VL53L0X 0x29, MCP-расширители)
- **Почему:** пока единственный I2C-девайс — OLED; отдельный владелец шины (io) — лишняя обвязка на старте
- **Цена:** при добавлении 2-го I2C-устройства шину надо расшарить (создать в `io`, передавать handle), иначе конфликт порта
- **Когда чинить:** при первом втором I2C-девайсе (этап 8/10) — вынести bus в `io`, `display`/сенсоры берут handle

### [T-005] Popup дисплея не именует новые enum-параметры
- **Где:** `components/display/src/display.cpp` (`draw_popup`, таблица `kWave`)
- **Что:** popup именует значения только `waveform` (id 2); `osc2_wave`/`osc3_wave`/`filter_mode` при смене показываются числом, не именем
- **Почему:** батч 3.1–3.4 добавил enum-параметры; спец-кейс именования пока только для осц1-формы
- **Цена:** мелкая — на отладочном OLED enum-значения читаются числом (0..3), а не словом
- **Когда чинить:** дёшево при желании (обобщить именование enum по id) или естественно на этапе 9 (цветной ST7796 + полноценный GUI); на функциональность не влияет

### [T-006] MIDI-вход: приходит только MM_MIM_OPEN, нот нет
- **Где:** `app/midi/midi_windows.go` (WinMM-транспорт), UI `app/ui/midibar.go`
- **Что:** на железе (Arturia Keystep 32) устройство открывается, но нот нет. Диагностика в баре дала
  `data=0 всего=1 id=03C1` — в очередь окна пришёл ровно один `MM_MIM_OPEN` (0x3C1), нот-сообщений
  (`MM_MIM_DATA` 0x3C3) — ноль. То есть **плумбинг верный** (WinMM постит в наше окно, механизм
  `CALLBACK_WINDOW` работает, поток правильный), но выбранное устройство не отдаёт нотные данные.
- **Почему:** транспорт WinMM написан вслепую (нет Windows+MIDI в облаке); отлаживали по встроенному
  счётчику. Дошли до «плумбинг ок, данных нет» и отложили по решению пользователя.
- **Цена:** MIDI-вход не работает; парсер (`app/midi/parse.go`) и UI готовы и под тестами. Пройденный путь
  (почему не `CALLBACK_FUNCTION`/`CALLBACK_THREAD`) задокументирован в шапке `midi_windows.go`.
- **Когда чинить:** вернуться позже. Гипотеза №1: Keystep выставляет **несколько** MIDI-входов
  (напр. «KeyStep 32» и «MIDIIN2 (KeyStep 32)»), а клавиши шлют на другой порт, чем открыли — проверить
  все имена из `List()` и открыть верный / попробовать каждый. Далее: сверить с эталоном (рабочий клиент),
  проверить, не занял ли порт другой софт, рассмотреть WinRT MIDI. Счётчик `всего/data/id` в баре — точка входа.

## Осознанные упрощения (не долг, а решение)

Записи здесь — не «надо починить», а «сделано намеренно, знаем о последствиях».

### [D-001] Проба пера на ESP32-S3, а не STM32
- **Что:** платформа выбрана из того, что есть, а не оптимальная для аудио
- **Почему:** цель — быстро пройти путь целиком, а не построить продукт
- **Цена:** слабее FPU, нет зрелой аудио-экосистемы, потолок по полифонии/эффектам
- **Когда чинить:** никогда в этом проекте. Следующая платформа — STM32 с нуля

### [D-002] Кнопки без velocity
- **Что:** матрица 4×4 на тактовых кнопках, динамика нажатия не читается
- **Почему:** velocity-keybed дорог и громоздок, противоречит компактности
- **Цена:** нет живой динамики от игры
- **Когда чинить:** не чинить. Velocity — как параметр шага (p-lock), путь Elektron

### [D-003] Монтаж без хедеров
- **Что:** модули паяются проводами напрямую, не съёмные
- **Почему:** хедеров нет в наличии
- **Цена:** замена сгоревшего модуля = распайка 5-7 проводов
- **Когда чинить:** если хедеры появятся — переделать на съёмные

### [D-004] Слои audio/io — заглушки
- **Что:** `control` наполнен на 0.2, `comm` на 0.3; `audio_init`/`io_init` пока только логируют
- **Почему:** DSP и периферия строятся на своих этапах; каркас держит структуру заранее
- **Цена:** нет — ожидаемое состояние; наполняются, а не «чинятся»
- **Когда чинить:** по roadmap (audio → этап 1, io → этап 8+)

### [D-005] Тулчейн: зеркало Espressif + libusb из apt, версия IDF запинена
- **Где:** `tools/setup-esp-idf.sh`
- **Что:** `IDF_GITHUB_ASSETS=dl.espressif.com/github_assets`, `libusb-1.0-0` ставится через apt, IDF пин `v6.0.2`
- **Почему:** egress облачной сессии режет `github.com/.../releases` (403) → берём с CDN Espressif; openocd без libusb рушит `install.sh`; пин версии = воспроизводимость сборки
- **Цена:** зависимость от доступности зеркала Espressif и apt в контейнере; ручной bump версии
- **Когда чинить:** не требуется; при недоступности зеркала — переопредели `IDF_GITHUB_ASSETS` или собирай локально

### [D-006] Протокол Serial — бинарный, не текстовый
- **Где:** `components/comm/`, `docs/serial-protocol.md`
- **Что:** протокол бинарный (кадры), хотя roadmap 0.3 рекомендовал текст «на старте»
- **Почему:** выбор пользователя — эффективность важнее читаемости (без `strtof`/`snprintf`-float на МК); GUI/скрипт всё равно парсят программно
- **Цена:** 0.3 не проверить голым терминалом — нужен `tools/serialtest.py`; чуть больше кода (кадрирование + CRC)
- **Когда чинить:** не чинить — осознанное решение. Логика вынесена в чистые `frame.*`/`protocol.*` и покрыта host-тестом, так что риск невелик

### [D-007] Отладочный OLED — вне спеки, временный
- **Где:** `components/display/`, `docs/hardware-oled-ssd1306.md`
- **Что:** SSD1306 128×64 (I2C 0x3C) добавлен по просьбе пользователя; в спеке дисплей — цветной ST7796 + тач (этап 9)
- **Почему:** нужен экран для отладки звука раньше, чем дойдут руки до ST7796; свой мини-драйвер без внешних зависимостей
- **Цена:** параллельный дисплейный код, который заменится/дополнится на этапе 9; спеку не трогали
- **Когда чинить:** этап 9 — цветной ST7796; OLED можно оставить как аппаратный debug-порт или снять

### [D-009] ADSR — control-rate (раз в блок), не пер-семпл
- **Где:** `components/audio/src/env.cpp`, применение в `voice.cpp`
- **Что:** огибающие продвигаются раз в аудио-блок (dt≈1.33 мс), VCA-усиление линейно интерполируется по блоку (VCF-cutoff — ступенькой на блок)
- **Почему:** так дешевле по CPU и масштабируется на полифонию (этап 3.5); пер-семпл ADSR не нужен
- **Цена:** мин. эффективная атака ≈ один блок (~1.33 мс); при loop с near-zero A/D — блок-рейт жужжание (~750 Гц). Зиппера нет (лерп VCA)
- **Когда чинить:** не требуется; если понадобятся сверхкороткие атаки — считать env-точки пер-семпл для VCA

### [D-010] Микшер: soft-clip в голосе + hard-clamp на мастере
- **Где:** `components/audio/src/voice.cpp` (soft-clip `y/(1+|y|)`), `audio.cpp` (кламп перед int16)
- **Что:** сумма 3 осц + шум + ring может выйти за [-1,1]; голос мягко клипует, мастер жёстко клампит
- **Почему:** не нормируем по числу осц (иначе один осц тихий); пользователь рулит уровнями, saturation — характер
- **Цена:** при высоких уровнях — сатурация (не чистая сумма); для суммы голосов (3.5) soft-clip на голос обязателен
- **Когда чинить:** не требуется; при желании чистого хедрума — глобальный лимитер после суммы голосов

### [D-011] Полифония: latch/loop-голоса держат слот; CPU-потолок 8 голосов
- **Где:** `components/audio/src/synth.cpp`, `sdkconfig.defaults` (-O2)
- **Что:** голос считается «свободным» только в `ENV_IDLE`; latched-дрон или зацикленная (loop) огибающая
  никогда не доходят до IDLE → занимают слот пула до снятия latch/loop. Смена `poly_voices` на лету —
  внутренний all-notes-off (граница режима). CPU на 8 голосах — потолок; митигация -O2 + рендер только активных.
- **Почему:** так проще и предсказуемо; steal-oldest всё равно уводит самый старый при нехватке слотов
- **Цена:** много latched-нот в поли могут занять все слоты; смена поли рвёт удержанные ноты
- **Когда чинить:** если упрёмся — лестница урезаний CPU (по нарастающей): пропуск нулевых осц-слотов (уже),
  `IRAM_ATTR` на `voice_render`/`filter_process` (нет flash-cache столлов), кубический сатуратор вместо
  деления в soft-clip, снижение MAX_VOICES. Замер на железе (serialtest: аккорды 1/4/8) решает.

### [D-012] Мод-матрица 4.1: свободнобегущие LFO, крупные шкалы приёмников, generic-GUI
- **Где:** `components/audio/src/lfo.*`, `voice.cpp` (мод-матрица), `app/layout/layout.go`
- **Что:** сознательные упрощения ядра модуляции v1:
  - **LFO свободнобегущие** — не ретригерятся по note-on и не key-sync (общие на все голоса, тик на Core 0).
    Фаза не привязана к ноте → на коротких звуках вобл «ловится» в случайной точке цикла.
  - **Шкалы приёмников фиксированы и крупные:** pitch ±2 окт на полную глубину (`mod[PITCH]*24` полутона),
    cutoff ±6 окт (домен `flt_env_amt`). Тонкое вибрато/трель требует малой `depth` (~0.02) — на кнобе −1..1
    это узкий диапазон у центра.
  - **GUI матрицы — обычные контролы** (8×{src,dst,depth} в блоке «Мод-матрица»), без наглядной таблицы/линий.
- **Почему:** LFO без key-sync проще и дешевле (глобальный тик, не на голос); фикс-шкалы — без лишних
  параметров range; generic-GUI — панель матрицы полируется отдельным под-этапом (4.3).
- **Цена:** нет key-syncّ-модуляций (напр. каждая нота с одинаковой фазой LFO); тонкая настройка глубины
  на крупной шкале неудобна; матрица визуально «плоская» до 4.3.
- **Когда чинить:** key-sync — если понадобится ретриг LFO по ноте (сделать LFO пер-голосным или сбрасывать
  фазу на note-on); шкалы — вынести range в параметр при нужде; GUI — **этап 4.3** (наглядная панель матрицы).

### [D-013] Wave-огибающая/морф 4.2: 4 базовые формы, без редактора точек, без gate
- **Где:** `components/audio/src/{waveenv,wavetable}.*` (`wavetable_sample_morph`), `app/layout/layout.go`
- **Что:** упрощения фичи «эволюция тембра»:
  - **Морф — по 4 встроенным формам** (sine↔saw↔square↔tri), не по multi-frame user-wavetable. `pos ∈ [0,3]`
    кроссфейдит соседей; полноценные загружаемые фреймы (спека §3) — позже.
  - **Wave-огибающая — 8 слайдеров-точек** без наглядного редактора кривой (перетаскивание узлов). В GUI это
    8 вертикальных фейдеров (мини-форма); редактор точек — полировка позже.
  - **Wave-env не gate-driven:** стартует с note-on, но нет release-фазы. loop=цикл, !loop=hold на p8. Для
    модуляции тембра это ок; «огибающая, отпускающаяся по note-off» — отдельная механика, не делали.
- **Почему:** 4 формы дают морф «из коробки» (band-limited таблицы уже есть, длина общая на mip →
  фаз-когерентно); 8 слайдеров — работающий минимум; gate у wave-env усложнил бы без явной пользы для тембра.
- **Цена:** нет произвольных wavetable-фреймов; форма wave-env редактируется по одной точке; wave-env не
  «выключается» по release.
- **Когда чинить:** multi-frame wavetable — когда дойдут user-таблицы (спека §3); редактор точек — полировка
  GUI; gate у wave-env — если понадобится release-модуляция.

## Закрыто

### [D-008] Wavetable без band-limiting → закрыт на 3.0
- **Было:** формы saw/square/tri — наивные (одна таблица на всю тесситуру), алиасинг на верхах.
- **Стало:** октавные mip-таблицы (`wavetable.cpp`), по одной на диапазон [F0·2^m, F0·2^{m+1}),
  гармоники до Найквиста верха диапазона. Наполнение аддитивным синтезом на старте (рекуррента
  Чебышёва — быстро). Выбор mip по частоте раз в блок (`wavetable_mip`), в семпловом цикле —
  дешёвый lookup без log2 (`wavetable_sample(w, phase, mip)`). Пул ~64 КБ во внутреннем DRAM.
- **Цена/остатки:** таблица mip band-limited по ВЕРХУ своего диапазона → низ диапазона чуть глуше
  идеала (стандартный компромисс «одна таблица на октаву», незаметно на слух). Механизм общий —
  переиспользуется для user-wavetable (спека §3) и Clean-режима. Lo-fi (3.4) пойдёт мимо band-limit.
- **Проверка:** host-тест `test_wavetable.cpp` (нет энергии выше расчётного лимита гармоник);
  на железе — высокая нота пилой/меандром без «звона» (serialtest: C7 saw / C6 square).
- **Грабли (важно на будущее):** генерация таблиц — **только float**. Первая версия считала в
  `double` → у ESP32-S3 FPU одинарной точности, double идёт через софт-эмуляцию
  (`__muldf3`/`__adddf3`/`__divdf3`), генерация отъедала ~15 с на старте и валила task-WDT.
  Переписано на float/FPU + предрасчёт коэффициентов гармоник вне горячего цикла → < 1 с. Правило
  для всего DSP на S3: никакого `double` в горячих путях (и `sinf`/`cosf`/`powf`, не `sin`/…).
